About

When understanding a new framework I always want to see the following:

  • A "brochure" site at the root including "about" *
  • Let a user log in
  • After login, see some kind of dashboard
  • Let the user log out, and see that the dashboard is no longer accessible.

How did I get here?

I followed these steps:

  1. Build a basic working site with a "pretend" login.
    • Read the router guide - important because it covers "guarded" routes
    • Study the plunkr on the above page. It has a lot going on.
    • Start with a new ng-cli-generated project
    • Study the plunkr code again and again, re-reading above links, integrating pieces as I went
  2. Added a basic HTTP service
    • I used this sample service which is mentioned in the Medium article below. It's a nice simple way to prove an API call with no auth required, create faux users, authenticate users and prove a restricted API call.
    • On the brochure I added callign the unauthenticated API just to prove it works
  3. Call a (pretend) back-end to authenticate and use JWTs.
    • Read medium piece on angular with authentication with a back-end (albeit a mocked out one)
  4. Call a real back-end to authenticate.

A note about static "brochure" sites

* I'm not saying that static brochure content "should" be provided by an angular site. So far I think a brochure site should be outside of Angular, and I'm a fan of Jekyll.

That said, I think seeing how the non-logged-in part of a site works is a key part of understanding a new framework.
